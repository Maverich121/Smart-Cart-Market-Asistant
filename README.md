## Overview

Smart Cart is a graduation project developed at **Başkent University, Faculty of Engineering** (Computer Engineering Department) during the 2023–2024 academic year under the supervision of **Assoc. Prof. Dr. Emre Sümer**.

The application aims to enrich the grocery shopping experience by enabling users to scan products with their phone camera, receive detailed nutritional and ingredient information, get allergy warnings, calculate daily calorie needs, and generate personalized recipes — all from a single mobile interface.
---

## Key Features

- **📷 Product Scanning** — Capture or upload a photo to instantly recognize fruits and vegetables using a YOLOv5-powered object detection model (94.8% validation accuracy).
- **🥗 Nutritional Information** — Access detailed product content including carbohydrate, protein, and fat values.
- **⚠️ Allergy Warnings** — The app cross-checks cart items against the user's personal allergen list and displays real-time warnings before adding items.
- **🍽️ Recipe Generation** — Enter available ingredients and receive a custom recipe generated by a fine-tuned GPT-2 language model, with allergen filtering and ingredient compatibility checking.
- **🔢 Calorie Calculator** — Personalized daily calorie needs calculated from age, height, weight, gender, activity level, and health conditions (diabetes, cholesterol).
- **🛒 Smart Cart Management** — Add, adjust quantity, and remove products; cart syncs in real time with Firebase Firestore.
- **👤 User Profiles** — Persistent user accounts with health preferences stored securely in Firebase.

**Component Connections:**

- **Database ↔ Mobile** — Real-time sync of user preferences, cart data, allergen lists, and calorie profiles via Firebase Firestore.
- **NLP ↔ Mobile** — Recipe generation requests and responses handled through the Flask server using HTTP POST.
- **YOLO ↔ Mobile** — Images are base64-encoded by the app, POSTed to the Flask `/predict` endpoint, and classified results are returned as JSON.
- **User ↔ Mobile** — User interaction through a Flutter-based UI/UX covering all app screens and features.

## Module Breakdown

### Machine Learning (YOLOv5)

The image recognition module uses **YOLOv5** (You Only Look Once) for fast, high-accuracy object detection. The model is trained and hosted via **Roboflow**.

**Supported Classes:**

*Fruits (23 classes):* Pineapple, Pear, Avocado, Honeydew Melon, Tomato, Galia Melon, Watermelon, Melon, Kiwi, Conference Pear, Red Apple, Lemon, Lime, Banana, Pomegranate, Nectarine, Pink Apple, Orange, Royal Gala Apple, Yellow Apple, Cherry Tomato, Peach, Green Apple

*Vegetables (19 classes):* Carrot, Zucchini, Brown Cap Mushroom, Asparagus, Red Pepper, Frying Potato, Cabbage, Eggplant, Leek, Vine Tomato, Yellow Pepper, Garlic, Cucumber, Onion, Sweet Potato, Orange Pepper, Potato, Green Pepper

**Model Performance:**

- Validation Accuracy: **94.8%**
- Training tracked via loss curves and confusion matrix analysis
- Common confusion: Lemon ↔ Mango (addressed in class balancing)

**Image Processing Pipeline:**

Images from the mobile app are sent as base64-encoded strings to the Flask server, resized to `1024×978`, decoded to JPEG, and passed to the YOLOv5 model. The top predicted class is returned as JSON.

---

### Flask Server & API

The backend server is built with **Flask** (Python) and exposed to the internet via an **Ngrok** HTTPS tunnel with TLS encryption.

**Endpoints:**

| Endpoint | Method | Description |
|---|---|---|
| `/predict` | POST | Accepts a base64 image, runs YOLOv5 inference, returns the top predicted class as JSON |
| `/tarif_uretimi` | POST | Accepts ingredient and allergen strings, runs the GPT-2 recipe generation pipeline, returns a structured recipe JSON |

**Security:** All traffic between the mobile app and the server is encrypted via TLS through Ngrok's HTTPS tunnel.

The mobile app is built with **Flutter** and connected to **Firebase** for authentication and database operations.

### Natural Language Processing (GPT-2)

The recipe generation module uses a **GPT-2 model fine-tuned on the RecipeNLG dataset** with custom special tokens for structured recipe output.

**Special Tokens:**

```
<RECIPE_START>, <RECIPE_END>
<TITLE_START>, <TITLE_END>
<INGR_START>, <NEXT_INGR>, <INGR_END>
<INSTR_START>, <NEXT_INSTR>, <INSTR_END>
<INPUT_START>, <NEXT_INPUT>, <INPUT_END>
```

**Generation Pipeline:**

1. User inputs ingredients and allergens in Turkish via the mobile app.
2. Ingredients are translated to English using `MyMemoryTranslator`.
3. Ingredients are normalized (lowercased, singularized, special characters stripped).
4. The system first generates a compatible ingredient list using `<INPUT_END>` as the stop token.
5. Allergen ingredients are filtered out from the generated list.
6. The recipe (title, ingredients with amounts, instructions) is generated using `<RECIPE_END>` as the stop token.
7. Calorie and gram values are looked up from the nutrition Excel database.
8. The final recipe JSON is translated back to Turkish (`ensure_ascii=False` preserves Turkish characters).
9. The structured recipe is returned to the mobile app and displayed.

**Ingredient Compatibility (Flavor Bible):**

The `flavor_bible_full.csv` dataset is used to find ingredients that pair well together. The dataset is augmented by mirroring all `main ↔ pairing` relationships, and the `find_compatible_ingredient()` function uses set intersection to identify ingredients compatible with the entire current ingredient list. Ingredients can be replaced interactively if the user wishes to substitute one item while maintaining overall dish compatibility.
